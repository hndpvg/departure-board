# 内部設計

## データフロー

```text
Place
  -> Source
    -> Filter
      -> Departure
```

現在の「羽田空港 第3ターミナル」は、京急と東京モノレールのSourceを選択するプリセットである。画面にはSource選択を表示せず、Placeに設定されたSourceの出発便を統合表示する。

## 責務

### Place

利用者がいる場所、または統合表示プリセットを表す。表示対象となるSource IDの一覧を持つ。

JSONではSourceの重複を避け、一般マージ時刻表で同じSourceを複数Placeから再利用できるよう、`sources`にはSourceオブジェクトではなくSource IDを保持している。

### Source

時刻表の取得元となる駅・停留所・方向を表す。事業者、路線、駅名、有効状態、Source単位のFilterを持つ。

現在は手入力JSONだが、将来APIや別ファイルからDepartureを取得する場合も、Source IDを接続キーとして使用する。

### Filter

Sourceから得たDepartureのうち、Placeの目的に合う便だけを表示対象にする。

現在の条件:

- `trainTypes`: 表示を許可する種別
- `destinations`: 表示を許可する行き先
- 条件が省略または空配列の場合、その項目では絞り込まない

### Departure

個々の出発便を表す。必ず`sourceId`を持ち、どのSourceから取得された便か追跡できる。

羽田第3ターミナルの空港プリセットでは、空港からの脱出判断を補助するため、任意の`arrivalInfo`を持てる。京急では品川・大門、東京モノレールでは浜松町の参考到着時刻を表示する。これは現時点では一般機能ではなく、値がないDepartureでは表示しない。

```ts
type ArrivalInfo = {
  stationName: string;
  arrivalTime: string;
  timeType: "arrival" | "departure";
  source: "train-detail" | "station-timetable" | "manual";
  confidence: "matched" | "probable" | "manual";
};
```

`arrivalInfo`は終点到着時刻を表すものではなく、利用者が複数の出発便を比較するための
reference stops的な補助情報である。表示対象はPlace・Source・trainTypeの目的に応じて選ぶ。

Departureは将来、京成本線と成田スカイアクセス線の経路・乗り場差などを表現するため、
任意の`routeName`、`trackGroup`、`platform`、`via`を持てる。現時点ではSourceを無理に
分割せず、実データ取得元で安定して判別できる情報だけを追加する。

`serviceNumber`は列車愛称付き列車の号数を表す任意の文字列である。UIでは`trainType`の
直後へ連結して、例として「スカイライナー105」「成田エクスプレス42」と表示する。

`trainType`は表示名およびFilter候補として使う。色分けは列車種別そのものではなく、
利用者の乗車判断グループを表す`displayCategory`で行う。Sourceの`trainTypeCategories`を
Providerが解決し、Departureへ`displayCategory`を補完する。同じ「特急」「普通」でも
事業者・Sourceごとに異なるカテゴリへ割り当てられる。

`operatorLabel`は発車標の行先上部へ表示する運行系統名であり、事業者を表す`operator`とは
責務を分ける。ProviderはDepartureに明示された`operatorLabel`を優先し、未指定時はSourceの
既定値を補完する。Filter、`trainType`、`displayCategory`の判定には使用しない。

## 一般マージ時刻表への拡張

追加が必要な主な要素:

- 複数Placeの定義と、Place選択UI
- Source検索・選択UIと、ユーザー独自Placeの保存
- 駅コード、事業者コード、方向、番線などの安定した識別子
- API、手入力JSON、GTFS等を共通Departureへ変換するprovider adapter
- 含める条件だけでなく、除外条件、方向、時間帯、曜日を扱うFilter
- 祝日、臨時ダイヤ、ダイヤ改正日を扱う運行カレンダー
- リアルタイム時刻、遅延、運休と予定時刻の区別
- Source設定とユーザー設定の永続化
- 一般マージ時刻表で到着情報を扱う場合の、対象駅選択と経路上の停車判定

## 現在の設計で将来問題になりそうな点

- `Place.sources`はSource ID配列であり、想定型の`Source[]`とは異なる。再利用性を優先した設計だが、型導入時は`sourceIds: string[]`への改名を検討する。
- `operator`と`line`がSourceとDepartureに重複している。API由来の列車ごとの路線変化を許容できる一方、不整合を防ぐ検証が必要。
- Sourceの`color`は表示都合の属性であり、規模拡大時は事業者・ブランド表示設定へ分離した方がよい。
- 行き先・種別は表示文字列の完全一致で絞り込んでいる。表記揺れや多言語化に備え、将来は安定IDで比較する必要がある。
- 現在の曜日判定は祝日を扱わず、午前4時を一律の営業日境界としている。事業者別の運行カレンダーが必要になる。
- 全Departureをブラウザで読み込んでいる。対象Sourceや期間が増えた場合、サーバー側での取得・フィルタ・ページングが必要になる。
- `arrivalInfo`は現在、羽田第3プリセット用の手入力補助情報である。一般化する場合は、列車の停車駅・着時刻データから生成し、対象駅が実際に経路上にあることを検証する必要がある。
## 成田第2ターミナルの京成Source分離

成田第2ターミナルでは、成田スカイアクセス線と京成本線を別Sourceとして扱う。
両者は経路、停車駅、ホーム、所要時間が異なり、利用者の乗車判断に直接影響するためである。

- `keisei-access-nrt-t2`: 成田スカイアクセス線。スカイライナー、アクセス特急など。
- `keisei-mainline-nrt-t2`: 京成本線。快速特急、特急、快速、普通など。

Sourceは「どの乗り場・経路から出発する便か」を表す。`displayCategory` はSourceとは独立し、
利用者が列車グループを色で識別するために使う。

発車案内の行先上部にはSource名を表示し、次の優先順位で表示名を決定する。

1. `source.displayName`
2. `source.line + source.stationName`
3. `source.operator + source.stationName`
4. `source.operator`

`routeName` はSource分離後も、取得元との照合、データレビュー、将来の経路表示に利用できるよう
Departureへ保持する。将来ホーム情報を追加する場合は、Departureの任意フィールド `platform` に
便別の値を投入し、必要になった段階でUI表示を追加する。

## MVP表示ルール

表示設定UIでは内部用語を利用者向けに置き換え、Sourceを「駅」、`trainType`を「種別」、
`destination`を「行先」と表示する。内部モデルと保存形式は変更しない。
駅見出しは`source.displayName`を優先し、未設定時は路線名と駅名を組み合わせる。これにより
表示設定一覧でも、どの駅の時刻表かを判別できる。

スマホ幅では端末OS側の時計表示と重複するため、アプリ内の現在時刻・日付は非表示にする。
操作部は運行区分、設定、更新を小さくまとめ、Departure Board本体を上に寄せる。
発車行では種別・行先を主情報として上段に置き、路線・駅は下段の補助情報として小さく表示する。
Departure Board上の路線・駅は、`source.shortName`または`source.line`と`source.stationName`を
別要素に分け、路線名と駅名を2段で表示する。

`arrivalInfo` は交通手段を比較するための参考地点情報である。参考地点の実到着時刻または実発車時刻を
取得・照合できた場合だけ表示し、取得できない場合は表示しない。固定加算による推定値は使用しない。
UIでは`timeType`に応じて、到着時刻に「着」、発車時刻に「発」を付ける。
参考地点時刻の下には、出発時刻との差分を分単位で計算し「（14分）」のように表示する。
この所要時間はUI上の補助情報であり、Departureデータには保持しない。

羽田第3のSource表示は利用者の判断単位に合わせ、次の簡潔な名称を使用する。

- `京急線　羽田空港第3ターミナル駅`
- `東京モノレール　羽田空港第3ターミナル駅`

スカイライナーと成田エクスプレスは、`serviceNumber` が存在する場合に種別名の直後へ
「号」を付けて表示する。

現在のdisplayCategory:

- `keikyu-fast`: 京急 快特・エアポート快特、オレンジ系
- `keikyu-express`: 京急 特急・急行、青系
- `monorail-airport-rapid`: 東京モノレール 空港快速、紫系
- `monorail-section-rapid`: 東京モノレール 区間快速、緑系
- `keisei-skyliner`: スカイライナー、紫系
- `keisei-access`: アクセス特急、オレンジ系
- `keisei-mainline`: 京成本線 快速特急・特急、緑系
- `jr-nex`: 成田エクスプレス、赤系
- `local-rail`: 普通・快速、グレー系
- `airport-bus`: エアポートバス、グレー系

## 参考到着地点の確定ルール

`arrivalInfo` は比較用の主要地点を表し、実際の終点を表さない。表示時刻は交通手段同士を比較するための
目安であり、遅延・道路状況・取得元の時刻定義によって実時刻と異なる場合がある。

- 羽田・京急線: 品川、大門。京急蒲田止まりなど参考地点へ到達しない便では表示しない。
- 羽田・東京モノレール: 浜松町。
- 成田・京成本線: 経路に応じて日暮里、押上、日本橋。
- 成田・成田スカイアクセス線: 経路に応じて日暮里、押上、日本橋。
- 成田・JR: 東京。成田エクスプレスも参考表示は東京のみ。
- 成田・エアポートバス東京・成田: 東京駅のみ。

UIは駅名と参考時刻の2段表示を維持する。
東京モノレールの浜松町は、京急の大門と同じ右側の参考地点スロットへ配置する。列数は増やさない。

## 参考地点時刻の由来

`arrivalInfo`の時刻は、標準所要時間の単純加算を使用しない。列車詳細から取得できる場合は
`train-detail`、参考地点駅時刻表から同一列車を照合できる場合は`station-timetable`として保持する。
公式資料を確認して手動補正した場合は`manual`とする。いずれも利用できない場合は出力しない。

同一列車の照合は、`serviceNumber`、`trainType`、`destination`、`routeName`、空港駅の
`departureTime`、停車駅順、参考地点側時刻を用いる。特にスカイライナーと成田エクスプレスは
`serviceNumber`を優先する。曖昧な照合結果は実時刻として扱わない。

現在の由来:

- 東京モノレール・浜松町: 列車別停車駅ページから取得した実時刻。
- 京急・品川／大門: 京急T7列車別停車駅ページから取得した実時刻。
- 京成・日暮里／押上／日本橋: 京成列車別時刻表から取得した実時刻。
- JR・東京: JR列車詳細ページから取得した実時刻。
- エアポートバス・東京駅: 公式PDFに記載された便別到着予定時刻。

スカイライナーと成田エクスプレスは`serviceNumber`が確定した便だけ参考地点時刻を表示する。
スカイライナーは公式専用時刻表の「号」列と空港第2ビル発時刻を照合し、全便の号数を確定する。

現在の実装済み参考地点は、浜松町、品川、大門、日暮里、押上、日本橋、東京、東京駅である。
スカイライナーは公式専用時刻表を`station-timetable / matched`として使用し、バスは公式PDFの
便別予定時刻を`manual / manual`として使用する。

UIは確認済み時刻がある場合だけ現在の2段表示を行い、画面下部で表示がない場合があることを明示する。

## データ取得と表示Filter

データ取得層は取得元に存在する列車を可能な限り全件保持し、表示対象の選別はFilterが担当する。
成田第2の京成系統では、快速・普通・通勤特急・モーニングライナー等もJSONへ投入する一方、
初期画面はSourceの`defaultFilter`で空港利用向けの種別に絞る。
羽田第3の京急は、都心・京成方面の主要行先に加えて、京成成田、芝山千代田、宗吾参道、
成田スカイアクセス線経由成田空港を初期表示対象に含める。成田第2の京成本線は快速特急、特急、
通勤特急、モーニングライナーを初期表示対象に含める。成田第2のJRは成田エクスプレスのみを
初期表示し、快速・普通はデータに保持したまま表示設定からONにできる。

成田エクスプレスは、同一号数の列車詳細で複数の終着駅が確認できる場合、
`新宿・大船`のように行先を結合して表示する。
参考地点の東京時刻は、同一号数内に複数の東京発時刻がある場合、最も早い時刻を表示する。


## 2026-06 成田第2ターミナルSource整理

成田第2ターミナルには、徒歩アクセス可能な補助的Sourceとして`keisei-higashi-narita-nrt-t2`を追加する。Source表示は「京成東成田線 / 東成田駅」とし、既存の成田スカイアクセス線、京成本線、JR成田線、エアポートバス東京・成田とは別Sourceとして扱う。

東成田駅Sourceは初期表示ONとするが、初期Filterでは`普通`と`快速`をOFFにする。`普通`と`快速`はデータには保持し、表示設定からONにできる。芝山千代田方面は今回の利用目的から外すため投入しない。

成田第2のエアポートバス東京・成田はデータと参考地点時刻を保持するが、初期表示ではOFFにする。鉄道中心の初期表示とし、必要な場合に表示設定でONにする。

Source識別ドットは`displayCategory`とは独立したSource色として管理する。成田スカイアクセス線はオレンジ、京成本線は青、京成東成田線は水色、JR成田線は緑、エアポートバス東京・成田は水色系とする。宗吾参道はデータには保持するが、成田の京成系Sourceの初期表示からは外す。

成田JRの参考地点は東京のみとする。成田エクスプレス、快速とも東京の参考時刻は右側スロットに表示し、新宿の`arrivalInfo`は生成しない。


## 2026-06 serviceType自動判定と参考地点時刻表示ルールの確定

serviceTypeの自動判定は、ユーザーがlocalStorageに保存した手動選択を最優先する。保存値が現在のPlaceで利用できない場合は自動判定へフォールバックする。自動判定では日本の祝日、振替休日、国民の休日を`holiday`とし、その後に土日を`holiday`とする。いずれにも該当しない場合は`weekday`とする。

`arrivalInfo`は同じ参考地点に到着時刻と発車時刻の候補がある場合、到着時刻を優先して1つだけ表示する。到着時刻が取得できない場合のみ発車時刻を表示する。所要時間表示は、実際に表示している参考地点時刻と出発時刻の差分から計算する。

東成田駅Sourceの表記は「京成東成田線」に統一する。成田のバスは初期表示OFFを維持し、宗吾参道と京成成田は初期表示から除外する。ただし、取得元に存在する対象便はデータとして保持し、表示設定からONにできる状態を維持する。
