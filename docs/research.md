# MVP 時刻表データ取得調査

調査日: 2026-06-06

## MVP方針決定（2026-06-06）

ODPTでは京急・東京モノレールの利用可能な時刻表データを確認できなかったため、データ取得の自動化はMVP後へ延期する。

MVPでは、羽田空港第3ターミナルの京急・東京モノレール時刻表を手入力JSONとして保持し、現在時刻以降の便を統合表示する。これにより、先にPlace単位の統合表示とユーザー体験を検証する。

- 時刻表JSONはアプリのリポジトリで管理する。
- 公開前に入力データの正確性と利用条件を確認する。
- 画面には予定時刻であること、遅延・運休を反映しないことを表示する。
- 将来のAPI接続に備え、画面がデータ取得元へ直接依存しない共通Departure形式を維持する。

## 結論

- 京急・東京モノレールともに、公式Webサイトで駅時刻表を閲覧できる。
- 両社が一般開発者向けに直接提供している、利用条件が明確な公開時刻表APIは確認できなかった。
- 公式Webサイトのスクレイピングは技術的には可能だが、公開サービスのデータ取得方法としては推奨しない。
  - 京急は、公式サイトの情報を事前許諾なく他サイト等へ転用することを控えるよう求めている。
  - 東京モノレールは、時刻データの無断転載・複写・電磁媒体等への加工を禁止している。
- 現在のMVPでは手入力JSONを使い、UIとマージ処理を先に検証する。入力データの正確性と利用条件を確認するまでは、動作確認用サンプルとして扱う。
- 自動データ取得を行う将来版では、京急・東京モノレールを収録することを契約前に確認したうえで、`駅すぱあと API` または `NAVITIME API` のようなライセンス済み商用APIを利用する構成を推奨する。

## MVPで必要なデータ

対象 Place は「羽田空港第3ターミナル」。出発方向として、空港から離脱する便だけを対象にする。

| 事業者 | 対象駅 | MVPで必要な方向 |
| --- | --- | --- |
| 京急 | 羽田空港第3ターミナル駅（駅番号 `KK16`） | 空港線 上り・京急蒲田方面 |
| 東京モノレール | 羽田空港第3ターミナル駅 | 上り・モノレール浜松町方面 |

最低限必要な属性:

- 運行日
- 発車時刻
- 事業者
- 路線
- 方向
- 列車種別
- 行き先

注意点:

- 平日、土曜、日曜・祝日または土休日のカレンダー判定が必要。
- 0時台の便を前営業日の深夜として扱うか、暦日として扱うかを統一する必要がある。
- 臨時列車、運休、遅延は通常の駅時刻表だけでは反映できない。MVPでは「予定時刻」であることを明記する。

## 京急の時刻表取得方法

### 公式Webサイト

京急の[羽田空港第3ターミナル駅ページ](https://www.keikyu.co.jp/ride/kakueki/KK16.html)から、方向別・曜日別の時刻表に遷移できる。

- 下り: 羽田空港方面
- 上り: 京急蒲田方面
- 曜日区分: 平日、土曜、日曜・祝日
- 駅番号: `KK16`

遷移先は `norikae.keikyu.co.jp` のHTML時刻表で、発車時刻、種別、行き先を表示している。駅掲示時刻表PDFと全線時刻表PDFも公開されている。

例:

- [京急 羽田空港第3ターミナル駅](https://www.keikyu.co.jp/ride/kakueki/KK16.html)
- [京急 全線時刻表](https://www.keikyu.co.jp/ride/kakueki/schedule.html)
- [京急Web 羽田空港第3ターミナル・上り平日](https://norikae.keikyu.co.jp/transit/norikae/T5?d=1&dw=0&pFlg=1&slCode=253-6)

### APIの有無

京急自身が一般開発者向けに提供する公開時刻表APIは確認できなかった。

公共交通オープンデータセンター（ODPT）は、登録後にAPIキーを発行し、鉄道時刻表等をAPIで提供している。ただし、公開情報だけでは京急の本番利用可能な駅時刻表データが現在収録されていることを確認できなかった。登録後に `odpt:Operator` と `odpt:StationTimetable` の提供状況、ライセンス、商用利用可否を確認する必要がある。

### スクレイピング判断

HTMLまたはPDFから取得することは技術的には可能。ただし、京急の[ウェブサイトご利用案内](https://www.keikyu.co.jp/website-guide.html)では、サイト情報の複製を私的使用の範囲に限定し、事前許諾なしの他サイト等への転用を控えるよう求めている。

したがって、許諾なしの定期スクレイピングと取得データの公開表示は推奨しない。

## 東京モノレールの時刻表取得方法

### 公式Webサイト

東京モノレール公式サイトでは、駅時刻表と列車ごとの詳細時刻表をHTMLで提供している。また、駅時刻表PDFも公開されている。

- 上り: モノレール浜松町方面
- 下り: 羽田空港第2ターミナル方面
- 主な曜日区分: 平日、土曜・休日
- 列車種別: 空港快速、区間快速、普通

例:

- [東京モノレール公式サイト](https://www.tokyo-monorail.co.jp/)
- [平日 羽田空港第3ターミナル駅時刻表PDF](https://www.tokyo-monorail.co.jp/news/pdf/timetable2024/haneda3_1.pdf)
- [列車別の詳細時刻表例](https://www.tokyo-monorail.co.jp/timetable/train/00/00004.html)

列車別の詳細時刻表には、列車種別、各駅の着発時刻、基準日が含まれる。駅時刻表だけで種別と行き先を十分に取得できない場合、列車詳細ページとの突合が必要になる可能性がある。

### APIの有無

東京モノレール自身が一般開発者向けに提供する公開時刻表APIは確認できなかった。

ODPTについても、公開情報だけでは東京モノレールの本番利用可能な駅時刻表データが現在収録されていることを確認できなかった。登録後のデータカタログとライセンス確認が必要。

### スクレイピング判断

HTMLまたはPDFからの取得は技術的には可能だが、東京モノレールの[利用規約](https://www.tokyo-monorail.co.jp/policy/)は、掲載データの複製・転用・転載・電磁的加工・送信・頒布・二次利用等を断っている。また、詳細時刻表ページにも時刻データの無断転載・複写・電磁媒体等への加工を禁止する記載がある。

したがって、許諾なしのスクレイピングと再表示は採用しない。

## 利用可能なAPI候補

| 候補 | 状況 | MVP適合性 | 確認事項 |
| --- | --- | --- | --- |
| 京急・東京モノレール公式API | 一般向け公開APIは未確認 | 低 | 両社への個別問い合わせ |
| [公共交通オープンデータセンター](https://developer.odpt.org/) | 登録・APIキーが必要。鉄道時刻表APIを提供 | 条件付き | 両社の収録状況、本番利用ライセンス、表示義務、商用利用可否 |
| [駅すぱあと API](https://docs.ekispert.com/v1/api/index.html) | 法人向け。鉄道駅時刻表と列車情報APIあり | 高 | 両社・対象駅・種別・行き先の収録、料金、キャッシュ・再表示条件 |
| [NAVITIME API 2.0](https://api-sdk.navitime.co.jp/api/specs/api_guide/transport_index.html) | 交通機関の時刻表取得APIあり | 高 | 両社・対象駅の収録、料金、レスポンス項目、キャッシュ・再表示条件 |
| ジョルダン法人向けAPI | 法人向け交通案内APIを提供 | 条件付き | 駅時刻表APIの提供範囲、両社収録、料金、利用条件 |

### 推奨するAPI選定手順

1. 駅すぱあと API と NAVITIME API に、対象2駅ではなく同一 Place 内の2ソースを統合表示するサービスであることを伝えて問い合わせる。
2. 次のレスポンス項目が取得できるか確認する。
   - 運行日
   - 発車時刻
   - 事業者
   - 路線
   - 方向
   - 種別
   - 行き先
3. データの保存期間、キャッシュ、加工、エンドユーザーへの再表示、出典表示の条件を確認する。
4. 無料評価版で同じ日時の公式時刻表と照合する。
5. コストと契約条件がMVPに合わない場合、両事業者へデータ利用許諾を問い合わせる。

## スクレイピングが必要か

**公開MVPでは不要かつ非推奨。**

商用APIまたは事業者許諾を得たデータを使うべきである。スクレイピングは、次の問題を同時に抱える。

- 公式サイトの利用条件に抵触する可能性が高い。
- HTML構造やURLの変更で壊れる。
- ダイヤ改正、臨時列車、曜日区分の変更を検知する仕組みが別途必要。
- PDF解析では色や記号に埋め込まれた列車種別を誤認しやすい。
- アクセス頻度制御、障害監視、データ検証が必要になり、MVPの本質ではない運用コストが増える。

許諾を得てスクレイピングする場合でも、ユーザーの画面表示時に毎回取得する構成にはしない。低頻度のバッチ取得、構造検証、差分検知、手動承認後の公開を行う。

## MVPの推奨技術構成

### 現在のMVP構成

- フロントエンド: HTML + CSS + JavaScript
- 時刻表: リポジトリ内の手入力JSON
- 配信: 任意の静的ホスティング
- 日時処理: ブラウザ側で `Asia/Tokyo` 固定
- データ処理: 現在時刻以降への絞り込み、複数事業者のマージ、発車時刻順ソート

外部APIとビルド環境を必要としないため、統合表示の価値を最短で検証できる。データ取得を追加するときは、JSON読込部分をprovider adapterへ置き換える。

### 自動データ取得を追加する場合の推奨構成

- フロントエンド兼API: Next.js + TypeScript
- ホスティング: Vercel等のNode.js対応環境
- データ取得: サーバー側からライセンス済み交通APIを呼ぶ
- キャッシュ: Next.js Data Cache、または小規模なRedis/KV
- 日時処理: `Asia/Tokyo` 固定。祝日・運行日カレンダーをデータ提供元に委ねられない場合は日本の祝日判定を追加
- 監視: API失敗率、最終取得時刻、データ件数ゼロを監視

ブラウザから交通APIを直接呼ばず、APIキーをサーバー側だけに保持する。

### データ取得レイヤー

事業者やAPIベンダー固有の形式を画面に漏らさないため、provider adapter を置く。

```ts
type Departure = {
  id: string;
  departureTime: string; // ISO 8601 with +09:00
  operator: "keikyu" | "tokyo-monorail";
  line: string;
  direction: string;
  trainType?: string;
  destination: string;
  isRealtime: boolean;
  sourceUpdatedAt?: string;
};
```

処理の流れ:

1. Placeから複数のSourceを解決する。
2. provider adapterで各Sourceの出発便を取得する。
3. 共通の `Departure` に正規化する。
4. 現在時刻以降に絞り、発車時刻順にマージする。
5. 直近の一定件数または一定時間分を返す。

### キャッシュ方針

- 静的な予定時刻表は日単位またはダイヤ改正単位でキャッシュ可能。
- 「あと何分」はキャッシュせず、クライアントで現在時刻との差分を計算する。
- API障害時は、契約上許される範囲で直近キャッシュを表示し、「最終更新時刻」を示す。
- 予定時刻表しかない場合は、遅延・運休を反映しない旨を表示する。

## 推奨するMVP進行案

### 第1段階: データ契約の確認

- 駅すぱあと API、NAVITIME API、ODPTの順で対象データと利用条件を確認する。
- APIレスポンスのサンプルを取得し、必要属性が揃うか検証する。
- 特に京急の直通列車の行き先・種別が正しく取得できるか確認する。

### 第2段階: 非公開プロトタイプ

- 権利処理済みの限定サンプルデータで、Place、Source、DepartureのモデルとマージUIを検証する。
- 日付変更、祝日、終電後、データなし、API障害をテストする。

### 第3段階: 公開MVP

- 契約済みAPIをprovider adapter経由で接続する。
- 出典、予定時刻であること、最終更新時刻を表示する。
- 公式時刻表との自動・手動照合をリリース手順に含める。

## 未解決事項

- ODPTで京急・東京モノレールの駅時刻表が本番利用可能か。
- 商用API各社で、列車種別と最終行き先まで取得できるか。
- APIデータを発車時刻順に統合・加工して表示することが契約上許可されるか。
- キャッシュ可能期間と、障害時の保存済みデータ表示可否。
- 祝日、臨時ダイヤ、年末年始ダイヤの扱い。
- 将来リアルタイム遅延・運休情報を追加する場合のデータソース。
